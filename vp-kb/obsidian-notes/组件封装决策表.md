# 🗂 组件封装决策表

|判断维度|标准|建议|
|---|---|---|
|**复用次数**|≥3次 → 值得封装  <br>1–2次 → 可先不封装|先满足当前需求，再考虑抽象；避免“为了封装而封装”|
|**接口复杂度（props数量）**|≤5–7 个 → 理想  <br>8–10 个 → 较复杂  <br>>10 个 → 过度封装|封装时保持接口简单，复杂配置可拆分成子组件|
|**业务耦合度**|低耦合 → 通用组件  <br>高耦合 → 强业务绑定|高耦合组件可封装为局部组件，不用追求全局通用|
|**可理解性**|使用者无需额外文档即可理解 → 正常  <br>需查文档/复杂配置 → 过度封装|保持组件直观，易读易用|
|**灵活性 vs 复杂度**|灵活性提升明显，复杂度增加小 → 合理  <br>复杂度提升大，收益小 → 过度封装|封装越通用越简单，避免“为未来所有可能情况封装”|
|**维护成本**|单点修改即可 → 合理  <br>修改一处需改多处 → 过度封装|关注组件维护成本，不要为了封装而增加修改难度|

---

# ⚡ 使用方法

1. **每次新需求或已有组件改造时**，对照表格逐条评估
2. **如果复用次数少、props多、耦合强 → 考虑拆分或降低封装程度**
3. **封装时保持“最小必要接口 + 高复用性”原则**
